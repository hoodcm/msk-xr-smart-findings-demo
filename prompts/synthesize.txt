## Synthesis Rules

### Entity Resolution

Group tagged findings into clinical entities using these rules, applied in order of priority:

1. **Degenerative findings** (taxonomy_ids MID1500–MID1640): Group ALL degenerative sub-findings for the same anatomy into a SINGLE entity. This rule takes priority over rule 3. Joint space narrowing, osteophytes, subchondral sclerosis, subchondral cysts, etc. are descriptors of one degenerative process — not separate entities, even when tagged with different taxonomy_ids (e.g., MID1510 osteoarthritis and MID1530 osteophyte_formation for the same anatomy MUST be merged into one entity). Use taxonomy_id MID1500 (degenerative_changes) or MID1510 (osteoarthritis) as the entity ID depending on source language. The extracted_text should consolidate all descriptors into one sentence.

2. **Clinically overlapping taxonomy IDs**: When multiple taxonomy IDs describe the same clinical phenomenon for the same anatomy, merge them into one entity using the most specific taxonomy ID. Do not create separate entities for different facets of the same process. Common overlaps:
   - Subsidence (MID1250) + hardware migration (MID1230) + cortical remodeling (MID1720) for the same prosthetic component → single entity under MID1250 (subsidence). Migration and remodeling are descriptors of the subsidence.
   - Periosteal reaction (MID1760) caused by hardware subsidence → fold into the subsidence entity, not a separate entity.
   - Arthroplasty component descriptors (screws MID1080, wire MID1090, plate MID1070) that are part of the arthroplasty construct → fold into the arthroplasty entity (MID1010). Only create a separate entity for a component when it is clinically independent of the parent construct (e.g., standalone cerclage cables not described as part of the arthroplasty).
   - **Hardware abnormality cascade**: Periprosthetic lucency (MID1240), hardware loosening (MID1220), and screw backout (MID1235) describing the same hardware construct are a clinical cascade (lucency is the observation, loosening is the diagnosis, backout is the consequence) — merge into a single entity under the most clinically advanced finding present. Use the most recent report's description as the canonical extracted_text, since these findings evolve over time.

3. **All other findings**: Group by EXACT taxonomy_id + anatomy + assertion match ONLY. All findings with the same taxonomy_id, anatomy, and assertion value represent the same clinical entity across time. Never merge findings with different taxonomy_ids unless explicitly covered by rules 1 or 2 above.

Do not create multiple entities for the same finding.

4. **No cross-laterality grouping**: Never merge Left and Right findings into one entity. "Left hip" and "Right hip" are always separate entities, even for the same taxonomy_id. The only exception is non-hardware findings explicitly tagged "Bilateral [region]" in the input. **Hardware must never use "Bilateral" anatomy** — if the input contains a "Bilateral" tagged hardware finding, decompose its instances into the corresponding Left and Right entities.

5. **Negative-assertion findings**: Negative-assertion findings (assertion: "negative") are resolved into entities using the same rules as positive findings. A "no periprosthetic fracture" across three reports becomes one negative entity with three instances. Group by taxonomy_id + anatomy + assertion — negative and positive findings for the same taxonomy_id and anatomy are SEPARATE entities.

6. **Hardware negative roll-up**: When a positive hardware entity (arthroplasty, fixation, etc.) exists for an anatomy, ALL negative-assertion findings for hardware-related taxonomy IDs (MID1200–MID1295) for that same anatomy MUST be merged into a single entity under MID1205 (hardware_abnormality) with name "No hardware abnormality" and extracted_text "No hardware abnormality." Individual hardware negatives (loosening, fracture, lucency, wear, subsidence) are clinical sub-checks of the parent construct — roll them up rather than enumerating separately. Instances from all merged negatives are combined chronologically.

### Composing extracted_text

The `extracted_text` is NOT a verbatim copy from the report. It is a **rewritten, standardized sentence** that you compose by:

1. Starting from the most descriptive instance (prefer descriptions with the most attributes and clinical detail, regardless of date)
2. Applying ALL phraseology standardization rules below to produce clean, normalized output
3. Applying status and stability language from the most recent instance
4. The result should read as a polished radiology sentence ready for clinical use

Example: if the source says "There are unchanged moderate degenerative changes of the right hip with osteophyte formation", the extracted_text should be "Similar moderate degenerative changes of the right hip with osteophyte formation." (perception statement removed, stability normalized, period added)

**For negative entities:** The extracted_text should be a clean affirmative statement of absence: "No periprosthetic fracture." "No hardware loosening." "Normal joint space." Remove hedging, perception statements, and qualifiers just as with positive entities.

### Temporal Ordering

- Sort instances within each entity chronologically (oldest first)
- When multiple reports describe the same finding, prefer the most recent complete description as the canonical extracted_text
- Preserve the original source_text verbatim for each instance (do not modify instance records)

### Phraseology Standardization Rules

Apply these rules to the extracted_text for each synthesized entity:

1. **Stability language**: Normalize "similar", "unchanged", "stable", "not significantly changed", etc. to begin with "Similar". Example: "Similar degenerative changes of the right hip." Never use "Stable" in the output.

2. **Remove perception statements and hedging language**: Strip "there is", "there are", "noted", "identified", "seen", "evident", "demonstrated", "present", "appears", "appearing", "suggest", "suggestive", "suggesting", "likely", "possibly", "presumably", "questionable", "compatible with", "consistent with", "may represent", "what appears to be", "probably", "suspected". Begin directly with the finding using definitive language. Chronic findings in the structured output are established facts — they must not contain hedging or uncertainty qualifiers.

3. **Remote timing**: Standardize chronic/remote language:
   - "Healed [location] fracture [with deformity descriptor if present]." — only for fractures with evidence of osseous healing
   - "Remote [location] [injury/change]." — for all other chronic posttraumatic findings: old injuries, avulsion sequelae, posttraumatic deformity, remote soft tissue injury. "Remote" is the default for non-fracture chronic trauma. Example: "old injury to the hamstrings origin" → "Remote right hamstring origin injury."

4. **Degenerative changes format**: Standardize to "[Severity if provided] degenerative changes of the [joint/structure]" with morphological descriptors. Consolidate multiple degenerative descriptors into one sentence per joint. Use the explicit `severity` field from the tagged input — do not re-parse severity from source_text. The severity field is already normalized to one of: mild, mild to moderate, moderate, moderate to severe, severe.

   **Spinal degenerative normalization**: Never use the term "spondylosis" in the output. Normalize spondylosis, degenerative disc disease, and all spinal degenerative sub-findings to "degenerative changes of the [spinal region]." Use taxonomy_id MID1500 (degenerative_changes) as the entity ID, not MID1570. The parent degenerative entity always takes precedence for naming.

   **Severity carry-forward**: Use the most recent non-null severity from the instance history. Degenerative changes do not regress radiographically. If a 2024 report states "moderate" and a 2025 report omits severity, the entity severity remains "moderate". Only update severity when a newer report explicitly states a different value. This rule applies to ALL entity types, not just degenerative changes.

5. **Hardware (lossless)**: Preserve all surgical descriptors exactly as provided. Start from the instance with the **most surgical detail** (component types, fixation methods, screw counts, cementation status) **regardless of date** — hardware configuration does not change between reports. Apply status and stability language from the most recent instance. Format: "[Status if stated] [laterality] [complete surgical construct], [additional component descriptors]." If any report in the series describes the construct as "revision", always include "revision" in the entity name and extracted_text — a revision does not stop being a revision.

6. **Fractures**: Format as "Healed [location] fracture [with deformity descriptor if present]."

7. **Calcifications/tendinopathy**: Format as "[Location] [type] calcification" or "[Location] calcific tendinopathy."

8. **Alignment**: Format as "[Alignment descriptor] alignment of the [structure]."

9. **Enthesopathy**: State as "[Location] enthesopathy."

10. **Heterotopic ossification**: State as "Heterotopic ossification [location]."

11. **Osteonecrosis**: Normalize AVN/avascular necrosis to "osteonecrosis". Use "Similar osteonecrosis of the [location]" if stability applies.

12. **Osteopenia**: Standardize bone density findings to "Osteopenia."

### Supersession Rules

**Rule 1 — Arthroplasty supersedes native joint findings:**
When a total joint arthroplasty is present for a given anatomy (e.g., "total hip arthroplasty"):
- Mark any degenerative changes, joint space narrowing, osteophytes, or osteoarthritis findings for that same joint as superseded=true
- Set superseded_by to the name of the arthroplasty entity (e.g., "Right total hip arthroplasty")
- Include these superseded findings in the output for the structured view, but they should NOT appear in the Report View

**Rule 2 — Revision supersedes prior hardware and all its abnormalities:**
When hardware is revised (revision arthroplasty, re-fixation, hardware exchange), ALL prior hardware abnormalities for that construct are superseded by default — the revision is intended to address them. This includes loosening, lucency, subsidence, backout, migration, varus/valgus angulation, periprosthetic fracture, hardware failure, and any other MID1200–MID1295 findings for that anatomy.
- Mark all pre-revision hardware abnormality entities as superseded=true
- Set superseded_by to the name of the revision entity (e.g., "Right revision total hip arthroplasty")
- The revision construct replaces the old construct — complications of the old construct are no longer active findings
- Only override this default if a post-revision report explicitly re-describes the same abnormality as persisting after the revision

**Rule 3 — Negative entity supersession:**
A negative-assertion entity can be superseded by a later positive-assertion entity for the same taxonomy_id and anatomy. For example, if "no joint space narrowing" (negative) exists from 2023 but "joint space narrowing" (positive) appears in 2025, the negative entity is superseded by the positive entity.

**Rule 4 — Negative suppression by active positive:**
When an active (non-superseded) positive entity exists for a given taxonomy_id and anatomy, do NOT create a negative entity for the same taxonomy_id and anatomy. The positive finding invalidates the pertinent negative regardless of temporal ordering. Example: if periprosthetic lucency (positive) exists from 2021, do not create a "no periprosthetic lucency" (negative) entity even if a 2022 report says "no periprosthetic lucency."

### Assertion Field

Each entity carries an `assertion` field ("positive" or "negative") from the tagged input. Positive entities describe findings that ARE present. Negative entities describe the documented absence of findings (pertinent negatives and normal observations).

Both positive and negative entities are first-class findings in the output. They follow the same entity resolution, phraseology, and supersession rules. There is no separate pertinent negatives mechanism — negative assertions are their own entities.

### Ordering Within Findings Array

Order the findings array deterministically using these sort keys in order:

**Primary sort — anatomy (alphabetical):** Group all findings for the same anatomy together (e.g., "Left hip" before "Lumbar spine" before "Right hip").

**Secondary sort — category priority (within each anatomy):** This follows clinical reporting convention — hardware and acute pathology assessments first, then chronic findings:
1. hardware
2. fracture
3. alignment
4. degenerative
5. soft_tissue
6. metabolic
7. osseous_lesion
8. osseous
9. inflammatory
10. postsurgical
11. variant

**Tertiary sort — assertion (positive before negative):** Within the same anatomy and category, positive entities sort before negative entities. This places each category's pertinent negatives immediately after its positives (e.g., hardware findings then "No hardware abnormality"; fracture findings then "No fracture or dislocation").

**Quaternary sort — entity name (alphabetical):** Within the same anatomy, category, and assertion, sort by entity name alphabetically.

This ensures identical input always produces identically ordered output, and the ordering mirrors standard MSK radiograph reporting convention.

## Output Format

Return a JSON object with a single array field `findings` containing zero or more SynthesizedFinding objects. Each SynthesizedFinding must have:
- name (string): normalized entity name, e.g., "Right total hip arthroplasty"
- taxonomy_id (string): e.g., "MID1010"
- taxonomy_name (string): e.g., "total_joint_arthroplasty"
- category (string): from taxonomy
- anatomy (string): e.g., "Right hip", "Pelvis"
- assertion (string): "positive" or "negative" — carried from the tagged input
- extracted_text (string): the synthesized, standardized, most-complete description
- severity (string or null): most recent non-null severity from the instance history — one of "mild", "mild to moderate", "moderate", "moderate to severe", "severe", or null if no instance ever stated severity
- stability (string or null): "similar" if applicable
- superseded (boolean): true if this finding was superseded by something else
- superseded_by (string or null): name of the superseding finding, if superseded=true
- instances (array of objects): each with date (string or null), source_text (string, verbatim from original), and assertion (string, "positive" or "negative")

If no findings result from synthesis, return { "findings": [] }
